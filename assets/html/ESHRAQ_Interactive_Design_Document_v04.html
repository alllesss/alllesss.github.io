01

Project Overview

ESHRAQ is a long-term artistic research project at the intersection of film, simulation, game technology, and philosophy. Inspired by the Illuminationist thought of the 12th-century Persian philosopher Suhrawardi, it explores knowledge through presence — a direct mode of understanding that emerges through experience rather than analysis.

The project exists across four output formats: an interactive simulation, a feature film (~75 minutes, planned for early-to-mid 2027), a multi-channel video installation, and an ongoing documentation blog at eshraq.io. All formats share the same underlying Unreal Engine simulation world.

This document describes the Interactive Simulation Prototype — the first public-facing interactive iteration of ESHRAQ, to be presented at IDFA DocLab in November 2026. It is a research prototype, not a finished production. Its purpose is to test how the ESHRAQ simulation functions as a public, embodied, interactive experience.

This prototype is not a game. It does not have objectives, menus, or a win condition. It is a simulation that responds to the presence of a body — activating, shifting, and returning to autonomy as people arrive and leave.

02

Experience Architecture

The core structural idea is that the cinematic film and the interactive simulation occupy the same screen and the same world. They are not two separate works — they are two states of the same system. The primary axis of the experience moves between these two modes: Cinema (the film plays, the world performs itself) and Simulation (a player is present, the world becomes responsive).

What drives the position on this axis is not a menu or a button — it is the physical presence of a person on the platform. No one present: the film plays. Someone steps on: the simulation activates. This transition is the central experiential proposition of the prototype.

Fig. 1 — Cinema / Simulation Axis

The cinematic film is structured as six narrative loops, each set in a different geographic location, each inhabited by a different soul. These same locations form the navigable worlds of the interactive simulation. The film and the interactive prototype share assets, environments, agents, and logic — they are two views into the same ontological system.

ESHRAQ — Film (20 min)

First two story loops  ·  vimeo.com/alllesss/eshraq

Within the simulation layer, the player navigates by selecting fragments from the film's timeline. Each fragment is a segment of the cinematic film that can be entered as a real-time interactive space. The player is not watching the film — they are inside the moment the film depicts, free to move through it according to the rules of their fragment type.

03

System States

The system operates in four states. Transitions between states are triggered by the physical presence or absence of viewers and players, detected by the platform sensor and the surrounding radar. No manual selection is required.

Fig. 2 — State Transition Diagram

AIdleno viewers · no player

The screen displays a field of grainy light shifting slowly between white, orange, and yellow. No film plays. No agents are active. The installation runs in an ambient mode, waiting for a body to enter the space. This is the baseline condition of the simulation before any presence is detected.

BViewers presentviewer detected · no player

When a viewer enters the radar's detection range, the grainy light field transitions toward the film. A red light bleed enters from the direction of the detected viewer. As the viewer moves closer, grain decreases and the film image sharpens. If they step away and no viewers remain, the video fades back to State A.

The viewer watches the film as a standard linear experience. A faint HUD overlay shows the radar output: number of detected viewers and their approximate positions relative to the platform. The system makes its own sensing visible without interrupting the film.

CSelectionplayer on platform · viewers may be present

When the player steps onto the platform, the HUD moves to the foreground and the film recedes behind the interface layer. The HUD displays the timeline: a horizontal sequence of fragment tiles representing segments of the cinematic film. Rotating the trackball scrolls through the timeline and highlights the active fragment. The player presses the Select button (right grip) to enter a fragment and proceed to State D.

If the player does not interact within a defined timeout window, the system automatically enters the currently highlighted fragment. Inaction is treated as a default selection.

DPlayplayer inside fragment · viewers may be present

The player is inside the selected fragment in real-time. The steering wheel handles directional navigation. Haptic feedback responds to surrounding geometry via rumble outputs on the left grip, right grip, and platform base. Each fragment is assigned one of four types, which determines the mode of presence available to the player. The Return button (left grip) exits the fragment at any time and returns to State C.

04

Fragment Types

In Play State, the player can play inside 4 different kinds of fragments. Fragments are similar to Game Mode in UE5 terminology. Each offers a different mode of real-time interactive experience. Each type is color-coded in the timeline UI, using the same visual logic as clip colors in a video editing application.

Fig. 3 — Timeline Fragment Selector (State C view)

Ghost

Free first-person navigation. The player moves through the simulated world as the soul, represented in-world as a ghost version of the physical interface object. The steering wheel controls horizontal direction and speed. The trackball controls the Z axis — rotating moves the player up or down through space. Agents run their loops and do not acknowledge the player. This is the primary exploration mode.

Player pawn rendered as interface object in-world. Z axis mapped to vertical translation via trackball. Haptics respond to surrounding geometry on all three rumble points.

Vessel

Partial agent possession. The player enters the perspective of a specific agent. The agent's AI loop continues but player input overrides it: the steering wheel redirects movement, the trackball switches between first-person and third-person camera. An optional subtitle layer delivers a monologue specific to the possessed agent — stored as text strings in a data table by agent ID, no voice recording required. The Return button exits possession.

Player input blended with AI navigation. Camera switch driven by trackball. Monologue content optional and independent of navigation logic.

Chaos

Destabilized first-person navigation. Movement is structurally identical to Ghost mode but the environment is distorted: visuals glitch, audio degrades, spatial orientation is disrupted. A single post-process intensity float drives both visual distortion and audio degradation. The trackball is the only relief — rotating it reduces intensity and partially clears the environment. Releasing allows chaos to reassert itself.

Movement system inherited from Ghost. Intensity float inversely mapped to trackball with return curve on release. No additional scene content required.

Witnessdeferred — phase 2

Constrained observation. The player orbits a fixed origin point derived automatically from the highest navigable point or centroid of agent activity. The steering wheel rotates the player around this origin. The trackball controls time of day globally via sky atmosphere and directional light. No manual per-scene origin placement required.

Excluded from V1 prototype. To be introduced once Ghost, Vessel, and Chaos are stable.

05

Hardware Interface

The physical interface is a custom-fabricated object built at MAKE Eindhoven. It consists of a platform base with a vertical column, a steering mechanism, and a top assembly housing the trackball and grip buttons. The interface is designed to be physically distinct from consumer game controllers — it communicates its own interaction logic through its form.

Steering wheel
Controls horizontal direction and movement speed in the simulation. Primary navigation input across all states and fragment types. One-axis input (rotation).
Trackball
Multi-purpose input whose function varies by state and fragment type. In State C: scrolls the timeline left and right. In Ghost and Chaos: controls Z axis (vertical position). In Vessel: switches camera between first and third person. In Chaos: reduces post-process intensity. One-axis input (rotation).
Select button
Located on the right grip. Confirms selection of the active timeline fragment in State C, transitioning the system to State D. Primary commit action.
Return button
Located on the left grip. Exits the current fragment in State D and returns to the timeline in State C. Cancel / back action.
Platform sensor
Detects whether a player is standing on the platform. Primary trigger for State C activation. Located in the base of the platform.
Radar sensor
Detects viewers in the surrounding space, including count and approximate direction relative to the platform. Drives viewer-presence logic in States A and B and feeds the HUD overlay. Mounted on or near the platform.
Haptic — left grip
Rumble output responding to objects or geometry on the player's left side in the simulation. Reinforces spatial presence and immersion.
Haptic — right grip
Rumble output responding to objects or geometry on the player's right side in the simulation.
Haptic — base
Low-frequency rumble in the platform floor. Reinforces major state transitions and significant in-world events.
RGB underlight
Broadcasts system state visually to the player and observers. Color and pulse pattern change with state. Fabricated into the base of the platform column.

07

Movement Model

Movement behavior differs across fragment types. The steering wheel is always the horizontal navigation input. The trackball's role changes depending on the active fragment. Across all types, movement is designed to feel physical and responsive rather than menu-driven.

Ghost
Steering wheel: heading + speed
Trackball: vertical Z axis
Constraint: none

Fully free 3D navigation. The player can rise above the world or descend into it.
Vessel
Steering wheel: override agent direction
Trackball: camera toggle (1st / 3rd)
Constraint: agent AI path asserts itself; deviation is possible but resisted.
Chaos
Steering wheel: heading + speed
Trackball: reduces distortion intensity
Constraint: none on movement, but environment is spatially disorienting.
08

Observer Experience

The installation is designed to be legible and compelling for viewers who are not on the platform. Observing someone else interact with the system — and watching the world respond — is a deliberate part of the experience. The work does not isolate the player; it makes participation visible to everyone in the room.

In State B, viewers directly influence what appears on screen through their proximity. Their approach causes the film to sharpen and the grain to reduce. They can see their own position reflected in the radar HUD overlay. This makes their presence as audience members visible and meaningful, even before anyone steps onto the platform.

In States C and D, the screen displays the simulation in real time. The RGB underlight changes color to reflect the active fragment type, broadcasting the system state across the room. Haptic events on the platform are accompanied by visible effects in the simulation, so observers can correlate physical and virtual action.

The radar HUD is visible throughout States B, C, and D, showing a live representation of how many viewers the system detects and where they are standing.

09

V1 Scope

The first prototype iteration targets IDFA DocLab, November 2026. Scope is constrained to ensure a stable, testable experience within the available development timeline.

Included in V1
  • All four system states (A, B, C, D)
  • Fragment timeline UI in State C
  • Ghost fragment type
  • Vessel fragment type (monologue optional)
  • Chaos fragment type
  • 2 loop environments (Ziggurat, Amsterdam)
  • Custom hardware interface (MAKE Eindhoven)
  • Platform presence sensor
  • Radar viewer detection
  • Haptic outputs (L grip, R grip, base)
  • RGB underlight state broadcasting
  • Data table–driven fragment configuration
  • State machine architecture in Unreal Engine
Deferred to later phases
  • Witness fragment type
  • Loops 3–6 (additional environments)
  • Vessel monologue audio (subtitles only in V1)
  • Multiple simultaneous players
  • Inventory or collectible systems
  • Dialogue trees or branching narrative
  • Menu-based UI of any kind
10

Technical Requirements

For the Unreal Engine developer

State machine
A clean state machine managing transitions between States A, B, C, and D. Each state owns its own logic. Transitions triggered by sensor events passed from the hardware abstraction layer.
Hardware abstraction
All physical inputs mapped through a single abstraction layer. Game logic references abstract input events, not hardware-specific signals. This allows hardware to be modified without touching game logic.
Fragment data table
A UE5 Data Table storing all fragment definitions: fragment ID, type (Ghost / Vessel / Chaos / Witness), entry transform, loop environment reference, and optional monologue data table reference. Fragment types are assigned here, not hard-coded in level Blueprints.
Fragment type system
Each fragment type is implemented as a self-contained mode. Ghost is the base movement system. Chaos inherits Ghost movement and adds a post-process intensity float driven by trackball. Vessel overrides AI navigation with player steering input and handles camera switching.
Timeline UI
HUD element displaying the fragment tile sequence in State C. Tiles are color-coded by fragment type. Active tile indicated visually. Driven by trackball input. Auto-selects on timeout if no input is received.
Radar HUD overlay
Persistent HUD element in States B, C, and D showing viewer count and approximate directional positions from radar input. Faint in States B and C, remains visible in D.
Gameplay tags
All zones, agents, and environment landmarks tagged via UE5 Gameplay Tags. Fragment configuration references tags rather than object names, enabling scalable architecture across multiple loop environments.

For the electronics collaborator (Mauricio)

Microcontroller
ESP32 or equivalent. Handles sensor reading, input processing, and serial / OSC output to Unreal Engine. Firmware programmed per agreed input/output protocol.
Communication
Serial or OSC over USB. All sensor values and input events transmitted as named float or boolean messages. Protocol to be agreed between developer and electronics collaborator before integration begins.
Sensor outputs
Platform presence (boolean), radar viewer count (integer), radar viewer directions (float array), trackball rotation (float, single axis), steering rotation (float, single axis), Select button (boolean), Return button (boolean).
Haptic outputs
Three independent rumble channels: left grip, right grip, platform base. Each receives an intensity float (0–1) from Unreal. Actuator type and mounting to be determined during fabrication at MAKE Eindhoven.
RGB underlight
Single RGB LED unit in the platform column base. Receives color (RGB float triplet) and pulse pattern from Unreal via the same communication channel as haptics.
